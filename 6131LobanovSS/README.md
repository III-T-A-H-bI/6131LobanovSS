# Лабораторные работы: Массовый поиск подстрок с использованием CUDA

## Описание проекта

В данном репозитории представлены две лабораторные работы, посвящённые ускорению алгоритма массового поиска множества подстрок в буфере с помощью технологий CUDA.

**Основные возможности:**
- Генерация тестовых данных (буфер `H` и набор подстрок `N`)
- Реализация алгоритма поиска на CPU (эталонная версия)
- Реализация алгоритма поиска на GPU с использованием **constant memory** для lookup-таблицы
- Сравнение производительности CPU и GPU
- Верификация результатов (полное совпадение с CPU)
- Визуализация результатов поиска (тепловая карта, статистика, выравнивание паттернов, диаграмма связей)

## Описание алгоритма

### Идея
Для каждого символа из буфера `H` мы должны обновить счётчики `R[i][pos]` для всех подстрок `i`, где этот символ встречается на позиции `k` внутри подстроки. Позиция начала возможного вхождения вычисляется как `pos = j - k`.

### Lookup-таблица
Для ускорения доступа к данным о символах всех подстрок используется таблица, отображающая байт `c` в список пар `(n_idx, k)`. Таблица размещается в **constant memory** GPU (64 КБ). Это обеспечивает низкую задержку и кэширование.

- Каждая запись упакована в 4 байта: старшие 16 бит – индекс подстроки `n_idx`, младшие 16 бит – позиция `k` внутри подстроки.
- Два массива в constant memory:
  - `d_offsets[257]` – начальные индексы для каждого байта (поиск по символу).
  - `d_lookup_data[MAX_LOOKUP_ENTRIES]` – упакованные записи.

### Ядро CUDA
- Каждый поток обрабатывает один символ буфера `j`.
- Поток получает диапазон записей для данного символа из `d_offsets`.
- Для каждой записи извлекаются `n_idx`, `k`, вычисляется `pos = j - k`.
- Если `pos >= 0`, выполняется атомарное увеличение `d_R[n_idx * h_len + pos]`.

### Интерпретация результатов
После завершения ядра массив `d_R` содержит для каждой подстроки и каждой позиции количество совпавших символов. Если для подстроки длины `len` в позиции `pos` значение равно `len`, то подстрока полностью совпадает с фрагментом буфера.

- `statistics.png` – гистограммы распределения длин, вхождений, топ-паттерны
- `chord.png` – диаграмма связей (распределение вхождений по паттернам и позициям)

## Критерии проверки для преподавателя

1. **Корректность алгоритма**
   - Результаты CPU и GPU идентичны для любых входных данных.
   - Программа `mass_search_test` гарантированно вставляет паттерны в буфер, и поиск их находит.

2. **Производительность**
   - Ускорение относительно CPU должно быть значительным (не менее 20x на реальных данных).
   - Использование constant memory оптимально (таблица не превышает 64 КБ).

3. **Визуализация**
   - Скрипт `visualize_results.py` должен корректно обрабатывать отсутствие файлов и генерировать все графики.
   - Отчёты должны быть информативными и наглядными.

4. **Автоматизация**
   - Скрипт `laba.sh` должен выполнять полный цикл без ошибок на чистой системе (после установки CUDA и Python).

5. **Стиль кода**
   - Комментарии на русском или английском, понятные имена переменных, отсутствие магических чисел.

## Возможные проблемы и их решение

- **Ошибка `MAX_LOOKUP_ENTRIES`**: Если программа сообщает о превышении размера constant memory, уменьшите количество паттернов или их длину.
- **Несовпадение результатов CPU/GPU**: Убедитесь, что вы используете `atomicAdd` и правильно интерпретируете матрицу `R`.
- **Пустые графики**: Проверьте, что файлы `buffer_H.bin`, `patterns_N.txt`, `search_results.txt` существуют и не пусты.
- **Ошибка `externally managed environment`**: При установке Python-пакетов используйте виртуальное окружение (скрипт `laba.sh` создаёт его автоматически).
